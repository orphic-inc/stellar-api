# ADR-0028: User-classes ladder and automated rank progression

**Status:** Accepted (2026-07-09)
**Date:** 2026-07-09
**Repos:** orphic-inc/stellar-api
**Relates:** [ADR-0006 — LinkHealth-gated ratio relief](0006-linkhealth-gated-ratio-relief.md) (the eligible-bytes semantics reused as the progression byte count), [ADR-0007 — CRS read-time value and event ledger](0007-crs-read-time-and-event-ledger.md) (the CRS-never-gates boundary this ADR extends to classes)
**Implements:** [PRD-10 — User Classes & Automated Progression](../prd/10-user-classes-and-automated-progression.md)

## Context

Stellar's `UserRank` model has always carried the level-gated privilege ladder (permissions, forum access, collage limits), but every rank change was a manual staff action through the admin panel. Epic #167–#171 shipped a fully automated version of that ladder: a `RankPromotionRule` schema and `User.rankLocked` flag (#167), a seeded ladder and rule set (#168), a background sweep job that promotes and demotes users and notifies them (#169), an admin criteria editor with a `rankLocked` override (#170), and a scoped-but-unshipped member-facing progress widget (#171, closed without a corresponding endpoint landing in this repo). The design decisions behind that epic — the classes-vs-CRS boundary, the byte-counting semantics, the prestige predicates, the demotion rules, the freeze guards — were settled during implementation but only ever recorded in a local, gitignored plan file (`USER_CLASSES_PLAN.md`) that was never checked into the repository and has since been retired. No ADR or PRD exists for a subsystem that materially changes a user's capabilities on its own, which is out of step with every other privilege-granting subsystem in this codebase. This ADR and its companion PRD close that gap by documenting the system as it actually shipped, reading the seed and the sweep job as the source of truth rather than the epic's issue text (which drifted from the final implementation in at least one place — see the note on notifications below).

## Decision

**The classes-vs-CRS boundary.** `UserRank` level is the privilege lever: it gates permissions, forum read/write/create thresholds, and collage limits. CRS (`src/modules/reputation.ts`) is a read-time reputation signal and stays strictly advisory. This mirrors the one-way layering already settled for Ratio in ADR-0006/ADR-0007/PRD-06 — a derived score can feed into CRS, but CRS never gates the mechanism it was derived from. The same firewall now applies to progression: no CRS dimension is a `RankPromotionRule` criterion, and no rank change writes back into a CRS dimension. If a future CRS dimension is ever proposed as a promotion input, that is a boundary change requiring its own decision, not a quiet addition to the rule set.

**The ladder, as seeded.** The seed (`DEFAULT_RANKS` in `src/modules/rankProgression.ts`, planted by `seedRanks`/`seedRankPromotionRules` in `src/modules/bootstrap.ts`) is authoritative over any prior issue text or schema-comment naming. The auto-managed ladder is: User (level 100) → Member (150) → Power User (200) → Elite (300) → Stellarific (350) → Stellartastic (400) → Stellarige (450). Staff (500) and SysOp (1000) sit above the ladder as `autoManaged: false` — they are staff-assigned only, never auto-reached by promotion and never auto-demoted into or out of. The three prestige tiers above Elite (350/400/450) grant no new permissions over Elite's; they exist purely as identity (color, badge, growing `personalCollageLimit`), a deliberate call recorded against the epic's own product decisions (#172).

**Rule shape and tuning.** One `RankPromotionRule` row is one rung: `fromRankId`/`toRankId`, `minContributed` (bytes, `BigInt`), `minRatio`, `minContributions`, `minAccountAgeDays`, an optional `extra` predicate, and `enabled`. Rows are runtime-tunable through the admin editor (`GET/POST/PUT /api/tools/promotion-rules`) without a deploy, guarded by `isAdjacentPromotionStep` so a rung can never be pointed past or around another level already on the ladder.

**Byte counting is link-health-eligible, not raw.** `minContributed` is checked against `getEligibleContributionBytes` (the same ADR-0006 relief pool: staff-approved, 72-hours-old, `linkStatus ≠ FAIL` bytes), not `User.contributed`. Progression therefore inherits ratio's "ongoing availability, not a one-time act" semantics — bytes earned from a link that later dies stop counting toward promotion the same way they stop counting toward ratio relief.

**The prestige predicate.** `RankExtraPredicate` has two values: `DISTINCT_RELEASES_500` (at least 500 distinct releases contributed to) and `QUALITY_CONTRIB_500` (at least 500 non-scene contributions that are either lossless or carry both a log and a cue). Seeded `DEFAULT_RULES` gates Elite→Stellarific on stock criteria only (no `extra`), then layers `DISTINCT_RELEASES_500` onto Stellarific→Stellartastic and `QUALITY_CONTRIB_500` onto Stellartastic→Stellarige, on top of the same stock numeric thresholds each rung already carries. They are pure data, evaluated by `extraMet` in the same pure evaluator as everything else.

**Evaluator purity.** `evaluateRankChange` (`src/modules/rankProgression.ts`) is pure and I/O-free by design, so promotion/demotion policy is unit-testable without a database. `rankProgressionJob.ts` is the thin DB-bound shell: it loads the ladder and rules, assembles each user's `RankProgressionInput`, calls the evaluator, and persists the result. This split is intentional and should be preserved by any future change to the ladder's logic.

**Demotion is symmetric on stock criteria only.** A demotion checks the same rule that promoted the user into their current rank, but only its "stock" fields — `minContributed`, `minContributions`, and the `extra` predicate. `minRatio` and `minAccountAgeDays` are excluded from demotion on purpose: a user never loses a class for ratio drift or simply because time has passed. One consequence of this design is the zero-consumption stall — because `computeRatio` returns `1.0` when `consumed` is `0`, a user who has never consumed anything cannot clear any rung requiring `minRatio ≥ 1.05` no matter how much they contribute; they stall at Member rather than being demoted, since they never promoted past it in the first place. Demotion is evaluated before promotion on every pass, so a user must still qualify for their current rank before the evaluator considers advancing them further.

**Freeze guards.** Two independent flags stop the evaluator from touching a user in either direction: `rankLocked` (an explicit staff freeze, toggled via `PUT /api/users/:id/rank-lock` — a route deliberately separate from `setUserRank`, because `setUserRank` replaces a user's entire secondary-rank set and would silently strip a Donor/VIP secondary on every automated change) and `hasActiveWarning` (any non-expired `UserWarning` freezes progression in both directions until it lapses or is cleared).

**Sweep mechanics and persistence.** `runRankProgressionSweep` batches ascending by user id (batch size 500) over active (`disabled: false`) users on an auto-managed primary rank, running hourly by default (`RANK_PROGRESSION_INTERVAL_MS`, default 3,600,000ms) after a 45-second startup delay, and advances or demotes each eligible user by exactly one step per pass — a property of the evaluator, not an artifact of batching. `applyRankChange` persists the decision with a direct `User.userRankId` update, never through `setUserRank`, for the same secondary-rank-preservation reason as the `rankLocked` route. The change is audited under the existing `user.rank_changed` action (tagged `auto: true`, with `direction` and `reason`), attributed to the lowest-id SysOp user as the system actor.

**Correction against the epic's issue text: notification, not private message.** The epic's tracking issue (#169) and the retiring plan file described the user-facing side of a rank change as a "System PM." The shipped code (`applyRankChange` in `rankProgressionJob.ts`) does not send a private message — it creates a `Notification` row (`NotificationType.rank_promoted` / `rank_demoted`), addressed through `SubscriptionPage.global_notices` with the new rank id as the page id. This ADR records the code's actual behavior; the "System PM" framing in the issue text should be read as superseded.

## Consequences

- The ladder's thresholds are governed by this ADR and by the seed in `bootstrap.ts`/`rankProgression.ts` going forward. The admin editor can retune numeric thresholds without a deploy, but adding, removing, or reordering a rung is a ladder-shape change that should be reconciled here, not just in the database.
- Classes and CRS remain fully decoupled. A future proposal to let a CRS dimension influence promotion (or vice versa) is a boundary change and needs its own decision, not a quiet addition to `RankPromotionRule`.
- Because promotion reads the same link-health-eligible byte pool as ratio relief, a user sitting near a promotion threshold can be pushed back below it by a link dying, with no action of their own — the same user-visible dynamic ADR-0006 already flagged for ratio, now also true of class progression.
- Issue #171's member-facing "progress to next class" widget was closed without an endpoint shipping in this repository — there is no gap-to-next-rank route today, and a member's only visibility into their standing is the rank itself. This is an open surface gap, not a design decision, and is left for a future issue rather than retrofitted here.
